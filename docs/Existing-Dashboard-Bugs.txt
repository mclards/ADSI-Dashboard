Existing Dashboard Bugs:
1. [FIXED 2026-04-25] 'Unexpected prior shutdown — the last run ended without any graceful shutdown signal (detected at 4/25/2026, 6:38:43 PM). Common causes: BSOD, forced power loss, or Windows killing the app past its shutdown budget. Check Event Viewer → System for Kernel-Power 41 and Application Error around that time to narrow the cause.' this always happen even I exited normally. Make sure during exit the shutdown is gracefull.
   ROOT CAUSE: Several silent `app.exit()` paths bypassed the normal shutdown chain (singleton-lock deny, license-startup-cancel, login-recheck-cancel, integrity-recovery-dialog-fail, license-activate-cancel) — they wrote a boot sentinel but never a matching shutdown-reason.current.json, so the next boot misclassified as "unexpected".
   FIX: electron/main.js — added `recordEarlyExitMarker()` helper and called it before every silent `app.exit()`. Also moved `readLastShutdownSync()` to AFTER the singleton-lock check so second-instance launches no longer overwrite the running first instance's sentinel + corrupt prev.json.
2. [FIXED 2026-04-25] Analytics 'LOAD VIEW' button did not forcely load updated data.
   ROOT CAUSE: Click handler called `loadAnalytics()` without `{force: true}`. (a) The `analyticsFetchInFlight` gate could swallow rapid clicks. (b) The 5-minute client-side Solcast est-actual cache (keyed by date) returned stale data on repeat clicks. (c) Browser ETag/304 validation could short-circuit the analytics endpoints when the server response was unchanged.
   FIX: public/js/app.js — click handler now passes `{force: true}`. In `loadAnalytics`, force=true: deletes `State._solcastEstCache_<date>`, appends a monotonic `&_t=<now>` cache-buster to every analytics API call (energy, dayahead, report/summary, legacy energy/5min) so HTTP-level caches are bypassed too.
